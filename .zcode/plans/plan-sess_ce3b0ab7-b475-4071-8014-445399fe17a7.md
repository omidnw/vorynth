## Implementation Plan

Three workstreams: (A) fix the citation bug + sources list, (B) build history backend, (C) build history drawer UI.

---

### Part A — Citation fix + sources list (small, high-impact)

The root cause is ONE regex that fails on comma-form `[1,3,5]`. Fixing it makes tooltips work AND populates the sources list (because `summary.citations` will no longer be empty).

**1. Backend parser** — `apps/core-engine/src/modules/intelligence/prompts/citations.ts`

- Change `CITE_RE` from `/\[(\d{1,3})\]/g` → `/\[(\d[\d\s,]*)\]/g` (matches `[1]`, `[1,3]`, `[1, 3, 5]`)
- Update `extractCitedNumbers()`: for each match, split the captured group on `,`, parse each as int, dedupe, preserve first-appearance order
- `resolveCitations()` is unchanged (positional lookup still works)

**2. Frontend parser** — `apps/desktop/src/components/ui/CitedText.tsx`

- Update `parseCitedText()`: same regex change; one bracket group can now produce multiple cite `Part`s (one chip per number)
- `CitationChip` and `CitationList` unchanged — they already render correctly once citations resolve

**3. Prompt update** — `apps/core-engine/src/modules/intelligence/prompts/summary.prompt.ts`

- Change the multi-cite instruction from `'[1][3]'` → `'[1,3,5]'` (comma-separated, single bracket) so the LLM's natural output matches what we parse

**Result:** tooltips show source/title/date on hover, clicking opens the article, and `CitationList` renders the sources panel under the summary (no separate work — it's already wired in `PeriodSummaryPanel`, just starved of data).

### Part B — History backend

**New tables** — add to `apps/core-engine/src/db/schema.ts` AND the DDL string in `apps/core-engine/src/db/migrate.ts` (the two are maintained in parallel per project convention):

- `search_history`: `id` (text PK), `query` (text), `mode` (text enum `keyword|ai`), `result` (text JSON — SearchResult or AskResult), `title` (text, defaults to query), `archived` (boolean int, default 0), `tokensUsed` (int, default 0), `hitCount` (int, default 0), `createdAt`, `updatedAt` (timestamp_ms)
- `brief_history`: `id` (text PK), `period` (text enum), `periodStart`, `periodEnd` (timestamp_ms), `result` (text JSON — PeriodSummary), `title` (text), `archived` (boolean int, default 0), `storyCount` (int), `createdAt`, `updatedAt`
- `app_settings`: `key` (text PK), `value` (text) — extensible key-value store; seeded with `search_history.record_ai=true`, `search_history.record_keyword=false`

**New HistoryModule** (`apps/core-engine/src/modules/history/`):

- `history.service.ts` — `recordSearchEntry()`, `listSearchHistory()`, `getSearchEntry()`, `updateSearchEntry(id, {title?, archived?})`, `deleteSearchEntries(ids[])`, `clearSearchHistory()`; mirror set for brief; plus `getSetting(key, default)`, `setSetting(key, value)`
- `history.controller.ts` — `GET /history/search`, `GET /history/search/:id`, `PATCH /history/search/:id`, `DELETE /history/search` (body: ids[]), `GET /history/brief`, `GET /history/brief/:id`, `PATCH /history/brief/:id`, `DELETE /history/brief` (body: ids[]), `GET /settings`, `PATCH /settings`
- Depends only on DbModule. Exported so SearchModule + IntelligenceModule can import it.

**Wire recording at the source** (so it survives navigation/closes):

- `SearchService.keyword()` → records to `search_history` if `record_keyword` setting is true
- `SearchService.ask()` → records to `search_history` if `record_ai` setting is true (default true)
- `IntelligenceService.summarizePeriod()` → always records the `PeriodSummary` to `brief_history` (this is the change that makes brief history durable — currently summaries are regenerated and discarded)

**Shared types** in `packages/types/src/index.ts`: `SearchHistoryEntry`, `BriefHistoryEntry`, `AppSettings`, `HistoryList`.

### Part C — History drawer (frontend)

**New feature module** `apps/desktop/src/features/history/`:

- `history-api.ts` — `apiFetch` wrappers mirroring the endpoints above
- `history-store.ts` — zustand store: `open`, `open()/close()`, `selectMode`, `selectedIds`, `viewingEntry` (for detail view). Survives route changes.
- `HistoryDrawer.tsx` — a right-side full-height drawer (`fixed right-0 top-0 h-screen w-[420px] z-50` + backdrop), styled with existing tokens (`border-outline-variant`, `bg-surface-container-lowest`, `shadow-lg`). Three sub-views:
  - **List view**: entries with a type badge (`[AI]`/`[KW]` for search, period + date-range for brief), title, time-ago, hit/source/story count. Click body → detail view. Hover/more-menu per row: Edit title, Archive, Delete.
  - **Detail view**: renders the cached result. For search AI → the answer with `CitedText` + `CitationList` + sources (reusing the Ask AI UI you like). For keyword → hit list. For brief → the `PeriodSummaryPanel` content. A "Re-run"/"Re-search" secondary action where applicable.
  - **Select mode**: checkbox per row, bottom action bar (Archive selected / Delete selected / Cancel).
- **Context-aware**: reads `useLocation()` — on `/brief` shows brief history; everywhere else shows search history (no tabs, per your choice).

**Header button** in `apps/desktop/src/app/ShellLayout.tsx`:

- New icon button (`history` Material Symbol) placed in the right-side header group, before the divider that wraps `ThemeToggle`. Same styling as ThemeToggle (`text-on-surface-variant hover:text-primary`). Calls `historyStore.open()`.

**Mount**: `<HistoryDrawer />` added next to `<JobsTray />` in ShellLayout so it persists across routes (only within the shell — not on onboarding/analyzing, matching JobsTray).

**Settings section** in `apps/desktop/src/pages/SettingsPage.tsx`:

- New "History" section with two toggles bound to `app_settings`: "Save Ask AI searches" (on by default), "Save keyword searches" (off by default). Persists via `PATCH /settings`.
